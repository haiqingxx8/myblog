JVM垃圾回收算法和经典垃圾回收器

发表于 2021-04-12 分类于 jvm垃圾回收 ， 垃圾回收算法 ， CMS, G1 阅读次数：
详细分析 Java 类加载机制，包括加载、连接、初始化阶段，以及方法区的作用。同时，探讨类初始化的触发时机、顺序和机制。

一、JVM垃圾回收的基础算法

所有GC收集器的设计都基于“如何高效识别垃圾并释放内存”这一核心目标，而基础算法则是实现该目标的“底层工具”。常见的三大基础算法各有优劣，直接决定了收集器的特性。

1. 标记-清除（Mark-Sweep）

  • 原理：分两步执行
    1. 标记：从GC Roots出发，遍历所有对象，标记出“存活对象”；
    2. 清除：遍历堆内存，回收所有未被标记的“垃圾对象”，释放内存空间。
  • 优点：实现简单，无需移动对象，回收速度快；
  • 缺点：会产生大量内存碎片（零散的空闲空间无法容纳大对象），后续分配大对象时可能触发频繁GC。
  • 应用场景：适用于对象存活率低、大对象少的场景（如CMS收集器的老年代回收）。

2. 标记-复制（Mark-Copy）

  • 原理：将堆内存划分为大小相等的两块（如From区和To区），仅使用其中一块（From区）：
    1. 标记：标记From区中的存活对象；
    2. 复制：将存活对象复制到To区，且紧凑排列（无碎片）；
    3. 切换：清空From区，下次GC时交换From和To的角色。
  • 优点：无内存碎片，对象分配时只需“移动指针”（bump the pointer），效率极高；
  • 缺点：内存利用率低（仅50%），复制大对象时开销大。
  • 应用场景：适用于对象存活率低的区域（如新生代，G1的Young GC）。

3. 标记-整理（Mark-Compact）

  • 原理：在“标记-清除”基础上增加“整理”步骤：
    1. 标记：标记存活对象；
    2. 整理：将所有存活对象向堆的一端移动，紧凑排列；
    3. 清除：回收对象移动后产生的连续空闲空间。
  • 优点：无内存碎片，内存利用率100%；
  • 缺点：需要移动对象，涉及引用地址更新，回收耗时较长。
  • 应用场景：适用于对象存活率高的区域（如老年代，G1的Mixed GC）。

二、CMS垃圾收集器：低延迟的“并发”实现

CMS（Concurrent Mark-Sweep）是JDK 8及之前版本中针对老年代的经典收集器，设计目标是“最小化STW（Stop-The-World）停顿时间”，适合对响应延迟敏感的应用（如Web服务）。

2.1 CMS的核心设计

  • 回收区域：仅负责老年代，需与新生代收集器（如ParNew）配合使用；
  • 核心算法：基于“标记-清除”，通过“并发”（与用户线程同时运行）减少STW；
  • 关键技术：三色标记算法（追踪存活对象）+ 增量更新（解决并发标记的漏标问题）。

2.2 CMS的完整回收过程

CMS的回收分为4个阶段，其中仅2个阶段需要STW，且停顿时间极短：

阶段 是否STW 核心工作
1. 初始标记（Initial Mark） 是（毫秒级） 标记GC Roots直接引用的老年代对象（如栈帧引用、静态变量引用），借道Young GC执行，停顿短。
2. 并发标记（Concurrent Mark） 否 与用户线程并发运行，从初始标记的对象出发，遍历老年代所有存活对象，标记过程无停顿。
3. 重新标记（Remark） 是（毫秒级） 处理并发标记期间用户线程修改的引用（如新增/删除引用），通过“增量更新”修正标记结果，停顿短。
4. 并发清除（Concurrent Sweep） 否 与用户线程并发运行，回收未被标记的垃圾对象，释放内存（基于“标记-清除”，产生碎片）。

2.3 解决并发漏标的关键：增量更新

并发标记时，用户线程可能修改对象引用（如“黑色对象（已处理）指向白色对象（未处理）”），导致白色对象漏标被误回收。CMS通过增量更新解决：

  • 触发时机：写后屏障（对象引用更新后执行）；
  • 逻辑：当黑色对象新增指向白色对象的引用时，将黑色对象重新标记为“灰色”，后续重新扫描其引用链，确保白色对象被正确标记。

2.4 CMS的优缺点

  • 优点：
    1. 低延迟：并发阶段无STW，适合Web等敏感应用；
    2. 响应快：STW阶段仅毫秒级，对用户体验影响小。
  • 缺点：
    1. 内存碎片：“标记-清除”导致碎片，大对象分配时可能触发“Concurrent Mode Failure”（并发回收未完成，老年代已满，退化为Serial Old收集器，导致长停顿）；
    2. CPU敏感：并发阶段占用CPU资源，降低应用吞吐量；
    3. 浮动垃圾：并发清除阶段用户线程产生的新垃圾，需下次GC回收。

三、G1垃圾收集器：分区化的“智能”回收

G1（Garbage-First）是JDK 9及以后的默认收集器，设计目标是“兼顾低延迟与高吞吐量”，尤其适合大内存堆（4GB以上）。其核心创新是“Region分区模型”和“收益优先”策略。

3.1 G1的核心设计

  1. Region分区模型：将整个堆划分为多个大小固定的Region（1MB~32MB，默认2048个），每个Region可动态扮演以下角色：

    • Eden Region：存放新创建的对象；
    • Survivor Region：存放Young GC后存活的新生代对象；
    • Old Region：存放晋升的老年代对象；
    • Humongous Region：存放超大对象（超过Region大小的50%），避免复制开销。

  2. 收益优先策略：GC时优先回收“垃圾比例最高”的Region（Garbage First），确保在用户设定的“最大停顿时间”（-XX:MaxGCPauseMillis）内，最大化回收效率。

3.2 G1的完整回收流程

G1的回收是“Young-Only阶段 → 并发标记周期 → Mixed GC”的循环，仅在必要时触发Full GC。

1. Young-Only阶段（Young GC）

  • 触发条件：Eden Region填满；
  • 过程（STW，短停顿）：
    1. 标记：从GC Roots出发，标记Eden和From-Survivor中的存活对象；
    2. 复制：将存活对象复制到To-Survivor（年龄+1）或Old Region（年龄达阈值-XX:MaxTenuringThreshold）；
    3. 清理：清空Eden和From-Survivor，将其归为“空闲Region”，To-Survivor变为下次的From-Survivor。
  • 核心算法：标记-复制（无碎片，新生代回收效率高）。

2. 并发标记周期（为Mixed GC准备）

  • 触发条件：堆内存使用率超过阈值（-XX:InitiatingHeapOccupancyPercent，默认45%）；
  • 过程（仅2个阶段STW）：
阶段 是否STW 核心工作
初始标记（Initial Mark） 是（借道Young GC） 标记GC Roots直接引用的对象，停顿短。
根区域扫描（Root Region Scanning） 否 扫描Survivor Region（根区域）中指向老年代的引用，需在下次Young GC前完成。
并发标记（Concurrent Marking） 否 与用户线程并发，遍历全堆标记存活对象（基于SATB算法，记录初始快照）。
最终标记（Remark） 是（毫秒级） 处理并发标记期间的引用变化（如删除引用），通过SATB修正标记结果。
清理（Cleanup） 部分STW 统计各Region的垃圾比例，排序后识别空闲Region（STW），清理RSet（并发）。

3. Mixed GC（混合回收）

  • 触发条件：并发标记周期完成后，老年代有可回收Region；
  • 过程（STW，可控停顿）：
    1. 选择CSet（回收集）：根据“收益优先”，选择垃圾比例最高的若干Old Region + 所有Eden/Survivor Region；
    2. 复制清理：将CSet中的存活对象复制到空闲Region（紧凑排列，无碎片），清空CSet Region并归为空闲。
  • 核心算法：整体标记-整理（老年代）+ 局部标记-复制（新生代），兼顾效率与无碎片。

4. Full GC（应急回收，需避免）

  • 触发条件：并发标记失败、晋升失败（老年代无空间）、堆内存不足；
  • 过程：单线程执行“标记-整理”，全堆扫描回收，STW时间极长（秒级），需通过调优避免。

3.3 G1的关键优化机制

1. RSet（Remembered Set，记忆集）

  • 问题：回收老年代Region时，若每次扫描全堆（如年轻代引用老年代），开销极大；
  • 解决方案：每个Region维护一个RSet，记录“外部Region对本Region对象的引用”（如Young Region引用Old Region）；
  • 效果：回收Old Region时，仅需扫描RSet记录的“来源Region”，无需全堆扫描，减少停顿。

2. SATB（Snapshot-At-The-Beginning，初始快照）

  • 问题：并发标记时，用户线程删除“灰色对象指向白色对象”的引用，可能导致白色对象漏标；
  • 解决方案：
    1. 并发标记开始时，记录全堆对象的引用快照；
    2. 写前屏障触发：当用户线程删除引用时，将删除的引用记录到SATB缓冲区；
    3. 最终标记阶段，重新扫描缓冲区中的引用，确保白色对象不被漏标。

3.4 G1的优缺点

  • 优点：
    1. 无内存碎片：Mixed GC通过复制实现紧凑排列；
    2. 可控停顿：支持-XX:MaxGCPauseMillis（软目标），适合大内存堆；
    3. 兼顾吞吐与延迟：并发阶段占用CPU少，Mixed GC灵活选择回收区域。
  • 缺点：
    1. 复杂度高：RSet、SATB等机制增加实现难度；
    2. 小堆性能差：Region管理开销在小堆（<4GB）上占比高，不如CMS。

四、GC算法的应用与实战优化

CMS与G1对基础算法的灵活应用，决定了它们的特性；而合理调优则能最大化其性能。

4.1 基础算法的应用对比

收集器 新生代算法 老年代算法 核心优化 内存碎片
CMS 标记-复制（ParNew） 标记-清除 增量更新+写后屏障 有
G1 标记-复制（Young GC） 标记-整理（Mixed GC） RSet+SATB+收益优先 无

4.2 实战优化策略

1. CMS调优（JDK 8及之前）

  • 避免Concurrent Mode Failure：
    • 设置-XX:CMSInitiatingOccupancyFraction=70（老年代使用率达70%触发CMS，默认92%，提前触发减少失败概率）；
    • 配置-XX:+UseCMSInitiatingOccupancyOnly，强制按阈值触发。
  • 减少内存碎片：
    • 开启-XX:+CMSCompactAtFullCollection（Full GC时执行整理，默认开启）；
    • 设置-XX:CMSFullGCsBeforeCompaction=3（每3次Full GC后执行一次整理）。
  • 监控重点：GC日志中的“Concurrent Mode Failure”“Full GC次数”“STW时间”。

2. G1调优（JDK 9及以后）

  • 控制停顿时间：
    • 设置-XX:MaxGCPauseMillis=200（根据业务需求调整，如Web应用设100~300ms）；
    • 避免超大对象：通过-XX:G1HeapRegionSize调整Region大小（确保大对象不进入Humongous Region）。
  • 避免Full GC：
    • 设置-XX:InitiatingHeapOccupancyPercent=40（提前触发并发标记，默认45%）；
    • 预留晋升空间：-XX:G1ReservePercent=15（默认10%，增加老年代预留空间，避免晋升失败）。
  • 监控重点：GC日志中的“Mixed GC次数”“Full GC次数”“Region垃圾比例”。

3. 通用优化原则

  1. 合理设置堆大小：新生代占比1/3~1/2（避免频繁Young GC），老年代预留20%以上空间；
  2. 减少对象创建：避免临时对象泛滥（如循环内创建字符串），降低GC频率；
  3. 监控GC日志：通过-XX:+PrintGCDetails打印日志，分析停顿时间、回收效率，定位瓶颈。

五、总结：如何选择收集器？

CMS与G1的适用场景差异显著，需根据业务需求选择：

场景 推荐收集器 理由
中小堆（<4GB）、低延迟优先 CMS 并发阶段无STW，STW时间短，适合Web等敏感应用（JDK 8及之前）。
大堆（>4GB）、兼顾吞吐与延迟 G1 无内存碎片，支持可控停顿，适合微服务、大数据应用（JDK 9及以后默认）。
极致低延迟（<10ms） ZGC/Shenandoah JDK 11+的新一代收集器，STW时间可忽略，适合金融、实时计算等场景。
