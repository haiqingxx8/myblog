Redis缓存实战

发表于 2025-09-11 分类于 架构 阅读次数：
覆盖常用缓存中间件、Redis特性、高可用方案、复制原理、持久化对比、雪崩/击穿/穿透与淘汰策略的工程落地与面试要点

Redis 缓存实战（面试与落地双友好）

1. 常用的缓存中间件

  • 本地缓存：Caffeine/Ehcache（纳秒级读取，进程内，容量受限，适合作为一级缓存）
  • 分布式缓存：Redis（最常用，丰富数据结构/脚本/事务/持久化/高可用）、Memcached（KV、内存高效、无持久化）、Tair/PolarDB-Cache/云厂商托管服务（免运维）
  • 典型分层：Caffeine（L1）+ Redis（L2）+ DB（源），读路径优先命中前两层

2. Redis 的核心特性（落到“能做什么、做得多好”）

  • 数据结构：String、Hash、List、Set、ZSet、Bitmap、HyperLogLog、Geo、Stream；支持原子操作
  • 高性能：单线程模型（命令串行避免锁冲突）+ I/O 多路复用（epoll/kqueue）+ 内存数据，P99 低毫秒级
  • 功能：TTL/过期、发布订阅、Lua 原子脚本、事务（MULTI/EXEC）、Pipeline、模块（Bloom、TopK 等）
  • 扩展：主从复制、哨兵（Sentinel）、集群（Cluster 16384 槽位）、RDB/AOF 持久化

3. 高可用实现方案（从简单到完备）

  • 主从 + 读写分离：主写从读，提升读吞吐；主挂需人工切换
  • Sentinel：自动主从故障转移（投票、仲裁、选主），适合小中型集群
  • Cluster：分片+高可用，每个主节点管理一部分槽位，节点间自动故障转移，水平扩展
  • 代理/网关：Twemproxy/Codis/云厂商网关，屏蔽分片细节、连接复用
  • 建议：中小业务优先哨兵；多租户/高并发/大数据量优先集群；配合代理降低客户端复杂度

4. 主从同步的实现原理（PSYNC）

  • 初次全量：从节点发送 PSYNC，主节点 bgsave 生成 RDB 并通过 socket 发送；期间主节点将新增写入追加到复制缓冲
  • 增量复制：基于 master_replid + offset 的复制积压(backlog)进行增量对齐
  • 断连重连：偏移量仍在 backlog 则部分重同步；否则回退到全量
  • 关键细节：复制缓冲区大小、磁盘/无盘复制（diskless）、min-replicas-to-write 与 min-replicas-max-lag 提升写入安全

5. 持久化方案对比

  • RDB（快照）：周期/手动 bgsave；优点：文件小、恢复快；缺点：崩溃时丢失最近一段数据
  • AOF（追加日志）：每次写入记录命令，重启回放；策略：always/everysec/no；优点：数据更完整；缺点：文件大、恢复较慢
  • AOF 重写：bgrewriteaof 将日志压缩为等价最小集，控制体积
  • 混合持久化（AOF + RDB 头）：启动更快、丢失更少（Redis 7 默认）
  • 实战建议：线上开启 appendonly yes；appendfsync everysec；定期重写；关键实例保留 RDB 以备快速恢复

6. 缓存雪崩 / 击穿 / 穿透

  • 雪崩（大量键同时失效→DB 被打爆）
    • 规避：TTL 加随机抖动、热点 key 提前预热、多级缓存、限流/降级、隔离核心与非核心业务
  • 击穿（单个热点 key 刚好过期→瞬时高并发打到 DB）
    • 规避：互斥锁缓存重建（SETNX + 过期 + 拿不到走旧值）、逻辑过期+异步刷新、后台单航道刷新(singleflight)
  • 穿透（查询本就不存在的数据→穿透缓存直打 DB）
    • 规避：布隆过滤器拦截、缓存空对象（短 TTL）、接口参数校验/鉴权、异常流量限速

7. 内存用尽与淘汰策略（maxmemory-policy）

  • 可选策略：
    • noeviction（默认）：内存不足直接报错
    • volatile-ttl/random/lru/lfu：仅淘汰设置了 TTL 的键
    • allkeys-random/lru/lfu：在所有键中随机或按 LRU/LFU 淘汰
  • 选择建议：allkeys-lfu（命中热度更稳）、配合 maxmemory 设置，控制采样样本（maxmemory-samples）
  • 相关参数：lfu-log-factor、lfu-decay-time 调整热度衰减；留 20% 安全余量，监控 evicted_keys

8. 常见缓存读写策略

  • Cache-Aside（旁路缓存）：应用先读缓存，失效后读库并回填；写先写库再删缓存（或双删）
  • Read/Write-Through：由缓存层代理读写并同步持久层（常见于托管/代理产品）
  • Write-Behind：异步批量写回降低写放大，注意数据一致性与丢失风险

9. 参数与运维速查（生产可用）

  • 持久化：appendonly yes；appendfsync everysec；auto-aof-rewrite-percentage 100；stop-writes-on-bgsave-error no（谨慎）
  • 复制与安全：replica-read-only yes；repl-backlog-size 按峰值写入与断连时间估；min-replicas-to-write 1；min-replicas-max-lag 10
  • 高可用：sentinel down-after-milliseconds 5000；failover-timeout 60000；parallel-syncs 1-2；cluster-node-timeout 5s；cluster-require-full-coverage yes
  • 资源：maxmemory 与 policy、tcp-keepalive、timeout、内核 somaxconn/文件描述符、numa 绑定与透明大页关闭
  • 监控：keyspace_hits/misses、used_memory/fragmentation、expired/evicted_keys、latency、commandstats、replication 健康

10. 面试速答模板（30s 结构化回答）

  • 问：Redis 高可用怎么做？
    • 答：小规模用 Sentinel（自动选主），大规模用 Cluster（分片+主从），读写分离+代理；关键参数（down-after、failover-timeout、node-timeout）+ 演练切换
  • 问：持久化如何选？
    • 答：AOF 保数据完整、RDB 恢复更快；线上混合持久化，everysec + 定期重写；业务分级决定数据可接受丢失窗口
  • 问：如何治理雪崩/击穿/穿透？
    • 答：抖动 TTL、预热、多级缓存、限流降级；互斥锁/逻辑过期/单航道刷新；布隆过滤、缓存空值、参数校验

11. 代码与项目实战场景（可直接复用）

  • 场景A：热点详情缓存（旁路缓存 + 逻辑过期 + 单航道刷新）
  • 场景B：排行榜与去重（ZSET + 布隆过滤器）
  • 场景C：分布式限流与滑动窗口
  • 场景D：购物车（Hash）与会话（String/TTL）

12. 缓存与数据库一致性（强/最终一致的工程策略）

  • 问题定义：缓存通常是“读高可用的副本”，写入链路中存在时间差或失败重试，导致“读到旧值/脏读/空读”。

  • 常见写策略与一致性影响：

    • 先写库再删缓存（推荐）：避免脏数据。若删失败→可能读到旧缓存；可通过重试/消息队列补偿或延迟双删缓解。
    • 先删缓存再写库：并发下可能读穿库再回填旧值，不推荐。
    • 更新缓存（写库同时更新缓存）：容易与并发写交错导致版本回退，不推荐。

  • 延迟双删（Delete-DB-DelayDelete）：

    • 流程：写库成功→删除缓存→延迟几十到数百毫秒再次删除缓存，覆盖并发回填的旧值。
    • 适用：写少读多、回填耗时短、允许小窗口不一致。

  • Binlog/CDC 驱动的订阅变更（强烈推荐）：

    • 通过 Debezium/Canal 订阅数据库变更→异步刷新或删除缓存，天然顺序、可重放。
    • 要点：按表/主键粒度构建缓存key，确保幂等；消费失败可重试或死信处理。

  • 事务消息 / Outbox 模式：

    • 业务库写入与“变更事件”写入同一事务的Outbox表，异步可靠投递消息（轮询或CDC）→消费者刷新/删除缓存。
    • 保证“有且仅有一次”的落地可通过幂等键与去重表确保。

  • 读写路径的工程化设计：

    • 读：Cache-Aside + 逻辑过期 + 单航道刷新，降低击穿；关键Key加热点保护；空值短TTL防穿透。
    • 写：以“写库+删缓存”为主；关键链路叠加CDC/Outbox；必要时启用延迟双删；高一致性场景加版本号/乐观锁。

  • 时序与边界讨论：

    • 并发交错：A删缓存→B回填旧值→A写库→旧值被保留；通过延迟双删或版本控制清理旧值。
    • 读旧值窗口：根据业务SLA设定可接受窗口（如<500ms），配合逻辑过期与异步刷新。
    • 淘汰与重建抖动：热点Key同步过期造成抖动，使用随机TTL和后台平滑刷新。

  • 示例代码：写库+删缓存+CDC 再保障

  • 运维与监控建议：
    • 指标：cache hit ratio、avg rebuild time、hot key QPS、inconsistent read rate（采样校验）、删除失败重试次数。
    • 告警：删除失败/延迟队列堆积、CDC 消费滞后、回放失败、热点异常。
    • 演练：并发写+读压测、掉 CDC 消费、延迟双删关闭与开启对比，度量不一致窗口。

结语：Redis 的工程化落地在于“分层缓存、正确高可用、恰当持久化、合理淘汰、完善监控与演练”。将参数、流程与演练固化为 Runbook 才能在高压场景下稳定可靠。
