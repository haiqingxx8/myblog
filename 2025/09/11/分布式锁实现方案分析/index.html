分布式锁实现方案与最佳实践

发表于 2025-09-11 分类于 架构 阅读次数：
系统性梳理分布式锁的需求与评估标准，横向对比 DB/Redis/Zookeeper/Etcd 等实现，给出参数与故障推演，并在文末从零手写一个可用的 Redis 分布式锁实现（支持解锁校验与可选续约）。

1. 为什么需要分布式锁

  • 多实例/多进程并发修改共享资源（库存扣减、定时任务单活、序列号生成）时，需要互斥保证。
  • 本地锁仅限单进程内，需引入跨进程的“协调器”。
  • 工程关注的不是“绝对强一致”，而是在成本、可用性、时延之间的合理权衡：在小概率故障下仍能快速恢复与自愈。

2. 评估标准（选型前自问）

  • 正确性：
    • 拿到锁的一定是当前“拥有者”（必须校验所有权再解锁/续约）。
    • 锁超时自动释放，避免永久死锁。
    • 释放只能由拥有者执行，避免“误删他人锁”。
  • 可用性：
    • 故障自愈（进程崩溃后锁能过期释放）。
    • 高并发低时延（获取/释放 O(1)）。
  • 扩展性：
    • 续约/看门狗，支撑长临界区。
    • 公平性/重入/租约与 Fencing Token 等增强能力。

3. 实现方案总览

  • 数据库基于行/唯一键：
    • insert 唯一键充当“锁占位”；定时清理超时记录；
    • 优点：依赖最小；缺点：重负载下 DB 成为瓶颈，删除/超时处理复杂。
  • Redis 基于 SET NX EX（推荐通用）：
    • SET lock value NX PX ttl 获锁；Lua 校验 value 后 DEL 解锁；看门狗续约。
    • 优点：简单高效、性能好；缺点：需正确处理时钟漂移、主从复制延迟、续约失败等边界。
  • Zookeeper 基于临时顺序节点：
    • 创建临时顺序节点，监听前一号节点删除来获锁；
    • 优点：天然会话语义、强一致目录；缺点：吞吐有限、实现偏复杂。
  • Etcd/Consul 基于租约：
    • 通过租约（lease）+ keepalive；以版本号实现 Fencing；
    • 优点：一致性强、API 完备；缺点：引入新组件与心跳开销。

4. Redis 分布式锁的关键细节

  • 正确加锁：
    • SET key value NX PX ttl（或 EX 秒）；value 用随机 ID（如 UUID-线程ID），避免误解锁。
  • 正确解锁（Lua 原子校验与删除）：
    • if get(key)==value then del(key) end；保证只由拥有者释放。
  • 续约（可选）：
    • watch dog 周期性将 TTL 延长；需在同一线程/拥有者上下文检查 value 一致。
  • 时钟与复制：
    • 单 Redis 节点通常足够；多副本读写需注意复制延迟，解锁与续约必须走主节点；
    • 若是跨机房或极高可靠，可用 Etcd/ZK 或在 Redis 上增加业务侧幂等兜底（见“Fencing Token”）。
  • 重入与公平：
    • 重入可在 value 中携带线程标识并在本地维护计数；
    • 公平性通常非刚需，若需要可通过排队队列实现。
  • Fencing Token（防止脑裂/超时误解锁后的旧持有者继续写）：
    • 每次成功获取锁发放单调递增 token，后端写入时校验 token >= lastToken，否则拒绝处理。

5. Zookeeper 方案要点

  • 临时顺序节点 + 监听前序节点删除事件，轮到自己时即获锁；
  • 会话断开节点自动删除，避免死锁；
  • 实现更重，但一致性与可观测性更好，适合强一致需求与 QPS 不高的场景。

6. 故障场景推演

  • 业务崩溃：锁 TTL 到期自动释放；若续约线程仍活，需以拥有者标识阻止误续约。
  • 解锁丢失：必须 Lua 校验 value；否则 A 可能删掉 B 的锁。
  • 长临界区：启用续约或按步骤拆分缩短锁持有时间。
  • 主从切换：只对主写，读也走主；切换期间失败要重试并回滚业务操作。

7. 选型建议（经验法则）

  • 首选 Redis SET NX EX + Lua，叠加监控与告警，绝大多数互联网场景足够。
  • 对强一致、强有序性更敏感：考虑 ZK/Etcd；
  • 低流量系统可用 DB 锁起步，但建议尽早迁移 Redis/Etcd。

8. 参数与监控建议

  • TTL：尽量覆盖 99.9% 临界区时长；如 5s～30s，配合续约间隔 1/3～1/2 TTL。
  • 获取等待：自旋 + 退避（10~50ms 抖动）直到 waitTimeout；
  • 监控：获取成功率、平均等待时长、持锁平均时长、续约失败率、同一 key 冲突率、误删告警（脚本返回值）。

10. 总结

  • 绝大多数场景，Redis SET NX EX + Lua 已足够；长任务叠加续约，关键写入叠加幂等与 Fencing，配合监控与告警，工程上既稳又省。
  • 对强一致要求高且 QPS 可接受，Zookeeper/Etcd 更安全；选型以团队运维能力与业务约束为主。

11. 市面主流方案解析与适用场景

在工程实践中，并不存在“一把锁走天下”，不同实现各有边界与最优解。下面按常见技术栈给出“原理速览 + 适用场景 + 工程要点”。

  • Redis + Redisson（高并发低时延的首选）

    • 原理速览：SET key value NX PX ttl 获取锁；Lua 校验 value 原子解锁；Redisson 在此之上提供重入、看门狗续约、Pub/Sub 通知等能力。
    • 适用场景：热点业务互斥（库存扣减单航道、定时任务单活、缓存重建）、对时延敏感、吞吐要求高的互联网场景。
    • 工程要点：
      • 所有权校验：value 存随机 ownerId，解锁必须 Lua 判断后再删，避免“误删他人锁”。
      • 长临界区：看门狗续约（周期 ~ TTL/3），续约前再次校验 ownerId。
      • 主从复制：获取/解锁/续约必须走主；如需提高复制可见性，可在关键路径叠加 WAIT N replicas。
      • RedLock 注意：对网络分区、时钟漂移敏感，非必要不建议；更推荐“单主 + 哨兵/集群 + 幂等 + Fencing Token”。

  • Zookeeper + Curator（强一致与可观测性的稳妥之选）

    • 原理速览：临时顺序节点排队 + Watch 前序节点删除获取锁；会话断开节点自动清理，避免死锁。
    • 适用场景：强一致、可观测性要求高且 QPS 可接受的后台任务、主从切换敏感的调度/选主类场景。
    • 工程要点：
      • 关注会话过期与重入语义；
      • 谨慎使用公平锁（吞吐下降明显）；
      • 结合 Curator recipes（InterProcessMutex、LeaderLatch）降低自研复杂度。

  • etcd/Consul（租约 + KeepAlive，具备天然 Fencing 能力）

    • 原理速览：通过 lease + keepalive 维持租约；etcd 以 revision/ModRevision 作为天然单调版本，可充当 Fencing Token。
    • 适用场景：跨机房、强一致、需要版本语义做“旧主隔离”的平台类系统（配置分发、控制面）。
    • 工程要点：
      • 连接心跳与租约续期的健壮性；
      • 将 revision 透传到下游写操作，拒绝旧版本写入；
      • 评估心跳与 RPC 成本，不建议在超高 QPS热点路径使用。

  • 数据库锁（唯一键/GET_LOCK/Advisory Lock）（低流量/低频任务的“就地可用”）

    • 原理速览：
      • 唯一键：插入占位行代表持锁，删除/过期释放。
      • MySQL GET_LOCK：基于连接的应用级锁，超时等待；
      • PostgreSQL Advisory Lock：会话或事务级别的整数锁。
    • 适用场景：内部管理工具、低频定时任务、小团队“无外部组件”诉求。
    • 工程要点：
      • 注意长事务与死锁风险；
      • 唯一键方案需清理过期占位；
      • PG Advisory 无 TTL，需要严格控制事务范围。

  • 关于“重入/公平/一致性”这些常被问到的点，应融入工程设计而非作为面试清单：

    • 重入：使用 Hash 计数或线程维度计数，解锁计数降至 0 再真正删除。
    • 公平：通过排队队列/票据可实现，但吞吐与复杂度代价高，除非强需求一般不推荐。
    • 一致性：锁只提供互斥，不等于业务一致性；务必叠加幂等、状态机/版本号、补偿（Outbox/Saga）。

12. 源码级解析：SET/NX/EX、Redisson RLock、Curator

  • Redis SET 语义

    • SET k v NX PX ttl：原子写入与过期设置；TTL 由服务端计时，避免客户端时钟漂移影响；DEL 非原子需配合 Lua 校验。
    • EVAL 脚本：在单线程执行模型下保证脚本内多条命令的原子性；返回值用于失败定位（如 0 表示未持有者）。
    • WAIT 复制确认：WAIT replicas timeout 可提升复制可见性，但会增加延迟，按需启用。

  • Redisson RLock 思路（简化描述）

    • 首次获取：Lua 尝试创建 Hash 结构保存线程持有计数；成功后启动看门狗续约（默认 30s）
    • 重入：同一线程二次获取直接计数 +1
    • 解锁：计数 -1；为 0 时删除 key 并发布解锁消息，唤醒阻塞端
    • 优点：工程化完善（续约、重入、Pub/Sub 通知），适合直接使用

  • Curator InterProcessMutex（ZK）思路（简化描述）

    • 创建临时顺序节点，监听前序节点删除；轮到自己即获锁
    • 会话断开自动删除节点避免死锁；吞吐有限但一致性强、可观测性好

13. 成熟方案的“真实案例”拆解（端到端）

为避免“只谈原理不落地”，这里选择三种在市面上广泛落地的方案，给出端到端案例与关键代码/配置要点。你可以直接作为蓝本替换业务细节。

  • 案例 A：Redisson 保证“订单关单任务单活”

    • 背景：订单系统需要每 5 分钟扫描超时未支付订单并关闭。应用多实例部署，只允许一个实例在同一时刻执行该批处理任务。
    • 方案：使用 Redisson 的 RLock，在调度任务开始前尝试获取锁，成功才执行；执行期间启用看门狗续约。
    • 关键代码（示意）：
      • 获取：RLock lock = redisson.getLock(“lock:job:close-order”); boolean ok = lock.tryLock(0, TimeUnit.SECONDS); if (!ok) return;
      • 执行：try { batchCloseOverdueOrders(); } finally { lock.unlock(); }
    • 关键参数：redisson.leaseWatchdogTimeout=30000；调度周期>=5min；任务分批（每批 100~1000）+ 断点续跑。
    • 监控：获取失败率、批处理耗时、持锁时长 P95、异常/重试次数。

  • 案例 B：Curator 实现“活动配置全局发布的选主与互斥”

    • 背景：运营活动发布流程涉及多步写操作（写缓存、刷新路由、推送配置），需要“有且仅有一位”发布者，且遇到发布失败能清晰回滚。
    • 方案：使用 ZK Curator 的 InterProcessMutex 控制发布互斥；同时利用 ZK 的可观测性（节点、watch 事件）做审计与排障。
    • 关键代码（示意）：
      • InterProcessMutex lock = new InterProcessMutex(client, “/locks/publish”); if (lock.acquire(3, TimeUnit.SECONDS)) { try { publish(); } finally { lock.release(); } }
    • 运维要点：ZK 三节点起步，关注会话过期；发布流程分阶段提交，失败可重试；监控前序节点/事件延迟。

  • 案例 C：etcd 租约 + 修订版号保障“配置中心的旧主隔离”

    • 背景：配置中心多个副本对外推送配置，需要确保任何时刻只有“当前主”能写；在主备切换后，旧主即使手握旧租约也不应继续写入。
    • 方案：利用 etcd lease + keepalive 维持主身份，同时依赖 ModRevision 作为 Fencing Token，在下游写入时强校验 revision。
    • 写入流程：
      • 竞选主：以租约写入 /leaders/config 的 KV；成功者为主，失败者 watch；
      • 推送配置：RPC 带上当前 revision；
      • 下游写库：校验 token >= lastToken，否则拒绝（保护旧主写）
    • 关键配置：租约心跳周期默认 5s；客户端与 etcd 的 gRPC keepalive；告警租约 keepalive 失败、leader key 抖动。

  • 案例 D：DB 唯一键“轻量互斥”应对内部工具的低频任务

    • 背景：内部报表计算任务每日一次；团队不希望引入外部组件。
    • 方案：创建 task_lock(task_name PK, expire_at)；插入占位代表持锁；失败则放弃；定时清理过期锁。
    • SQL 草图：INSERT INTO task_lock(task_name, expire_at) VALUES(‘report’, now()+interval ‘5 min’);
    • 风险控制：加唯一键防并发；执行完成删除占位；异常下过期释放；确保任务幂等（可重跑）。

  • 公共实践（跨方案复用）

    • 命名：lock:{biz}:{key}，ownerId=实例ID-线程ID-UUID（或租约ID）
    • 续约：周期=TTL/3；失败停止任务并报警；
    • 幂等：以业务流水/版本号兜底，锁只是互斥，不保证最终一致性；
    • 可观测性：埋点“获取/释放/续约/冲突/等待时延”，出现“误删他人锁”立刻告警与快照。
