分布式系统选举：Raft 算法（从原理到生产落地）

发表于 2025-02-07 分类于 架构 ， 分布式一致性 阅读次数：
以“选举”为切入点系统讲解 Raft：原理、异常场景、工程参数、监控与演练、面试问答与真实案例，并附上可落地的实现清单与伪代码思路。

1. 为什么需要选举（问题模型）

  • 分布式复制需要一个写入序：避免多个写主相互覆盖与脑裂。
  • 在 Raft 中，Leader 负责接收写请求并复制日志到多数派，Follower 只接受 Leader 的指令。
  • 选举确保“同一任期（term）内恰有一个 Leader”，并在 Leader 失联时快速产生新 Leader 以恢复可用性。

2. Raft 快速总览（本文聚焦选举）

  • 角色：Follower、Candidate、Leader。
  • 任期 term：单调递增，贯穿投票与日志复制，用于判断“新旧权威”。
  • 日志：条目包含 index 与 term，领导者保证“前缀匹配”后追加复制。
  • 本文重点讲“Leader 选举”，日志复制只在需要时简述其对选举的影响。

3. 选举流程（核心机制）

  1. Follower 在 election timeout 内未收到 Leader 心跳（AppendEntries）即转为 Candidate；
  2. Candidate 自增当前 term，给自己投票并并行向所有节点发送 RequestVote 请求；
  3. 节点授票条件：
    • 候选人的 term 不小于自身；
    • 候选人日志“至少不比自己旧”（lastLogTerm 较大，或相等时 lastLogIndex 较大）。
  4. 候选人获得多数派选票后成为 Leader，立即以空心跳（空 AppendEntries）宣告权威并复位 follower 计时器；
  5. 若出现平票（split vote）或投票被拒，等待下一次随机超时重新发起选举。

4. 关键参数与建议

  • 选举超时（Election Timeout）：随机化窗口（如 150~300ms、300~600ms，仅示例，须结合 RTT/GC/磁盘特性评估），不同节点应独立均匀采样以降低竞争。
  • 心跳间隔（Heartbeat Interval）：通常为选举超时的 1/10~1/3，确保 follower 定时刷新计时器；过短浪费带宽，过长则误判。
  • 磁盘与 GC：长时间停顿可能导致 leader 心跳中断与错误重选，需配合 GC 调优与 I/O 监控。

5. 异常与边界场景推演

  • 分票（Split Vote）：多个节点同时超时成为 Candidate，彼此瓜分选票，导致连续重试；应对：更大/更宽的随机超时窗口、PreVote、网络抖动下的退避。
  • 网络分区：
    • 多数派侧产生新 Leader，少数派旧 Leader 无法获得多数确认，继续写入将因复制失败而回退。
    • 分区恢复时，旧 Leader 见到更大 term 的心跳必须降级为 Follower。
  • 候选人日志落后：落后者应被拒绝授票，避免旧日志当选导致提交性违背；因此 RequestVote 比较 lastLogTerm/index。
  • 时钟与停顿：系统暂停（GC、容器抢占、I/O 阻塞）可能触发误判；需要更稳健的 timeout、CheckQuorum 与只读租约优化读路径。

6. 工程化加固清单

  • PreVote：在真正增加 term 之前做“预投票”，仅当多数节点认可它可能当选时才进入正式选举，避免抖动期频繁提升 term 干扰现任 Leader。
  • CheckQuorum：Leader 周期性确认自己仍能联系到多数派，否则主动降级，减少脑裂风险。
  • 持久化：currentTerm、votedFor、日志元数据必须同步落盘；重启后不能“忘记”投过票与任期。
  • 单线程状态机/串行化 RPC 处理：简化并发；或以严格锁保护状态切换与计时器。
  • I/O 隔离与限流：复制与快照不应阻塞心跳；网络风暴时保护“控制面”。

7. 参考伪代码（思路）

  • 定时器：
    • heartbeatTicker: 每 heartbeatInterval 触发 Leader 发送 AppendEntries；
    • electionTimer: 在 [minTimeout, maxTimeout] 随机重置，超时触发 startElection()。
  • 关键处理：
    • startElection(): term++、vote=self、向 peers 发送 RequestVote 并统计多数；
    • onRequestVote(req): 若 req.term < currentTerm 拒绝；若尚未投票且对方日志不旧则授票并重置计时器；
    • onAppendEntries(hb): 若 hb.term >= currentTerm 接受并降级/保持 follower，重置 electionTimer。

8. 与日志复制的耦合点

  • 新 Leader 当选后必须先通过心跳建立权威（带上当前 commitIndex），再通过 AppendEntries 同步落后者。
  • 候选人的“日志新旧”判断直接影响授票，避免旧日志当选。

9. 面试高频问答（含要点）

  • 为什么要对选举超时做随机化？
    • 避免同时超时导致分票，提高首次选举成功率。
  • 如何避免脑裂？
    • 多数派机制 + CheckQuorum；旧 Leader 见到更大 term 必须降级；只读用租约控制。
  • Raft 如何确保不会把票投给“日志更旧”的候选人？
    • RequestVote 比较 lastLogTerm/index，落后者被拒绝。
  • 为什么引入 PreVote？
    • 减少在网络抖动/少数派环境中无意义地提升 term 干扰集群稳定性。
  • Election Timeout 与 Heartbeat Interval 如何选？
    • 基于 99 分位 RTT 与系统停顿特征评估；通常 heartbeat << election，避免误判且不致过多占用带宽。

10. 生产参数与容量规划建议

  • 延迟基线：以集群节点间 RTT p95/p99 为基础，选举超时不少于若干倍；容器化/云盘需放大裕量。
  • 节点规模：节点越多，复制与快照开销越大，心跳扇出变宽；常见 3/5 节点，优先奇数。
  • 磁盘策略：WAL/日志 fsync 性能直接决定复制时延；隔离数据盘与系统盘，观察 queue depth。

11. 监控与告警（可观测性）

  • 角色与任期：当前角色、term、leaderId；Leader 切换频率（短时抖动告警）。
  • 选举耗时：从失联到当选的时间分布；选举失败/分票计数。
  • RPC 成功率与延迟：AppendEntries/RequestVote 成功率、心跳往返时间。
  • 日志复制健康度：跟随者落后程度（lag）、快照触发频率、磁盘写放大。

12. 故障演练与排障流程

  • 演练脚本：
    • 模拟网络分区（仅让 1~2 节点互通）、限速与抖动；
    • 注入 GC 停顿/CPU 抢占；
    • 放大磁盘时延（fsync 慢、队列堆积）。
  • 排障路径：
    • 先看 term 与角色变化是否异常频繁；
    • 结合 RPC 延迟、心跳丢包与 I/O 指标定位瓶颈；
    • 检查 election 超时是否过短导致误判；
    • 必要时开启/调整 PreVote、CheckQuorum、随机化窗口与退避策略。

13. 真实落地案例（蓝本）

  • 配置中心/注册中心（基于 etcd/raft）：
    • 3 节点主集群 + 2 节点只读从集群（非共识），选举超时 300~600ms，心跳 100ms；
    • 打开 PreVote/CheckQuorum，Leader 切换告警阈值 3 次/小时；
    • 只读接口使用只读租约，写入必须经 Leader。
  • KV 存储（Java/SOFAJRaft）：
    • 业务写入经 Leader 聚合，Follower 应用日志后暴露只读；
    • I/O 隔离心跳与复制线程，慢 Follower 自动限速/旁路快照。

14. 实施清单（Ready-to-Run）

  • 必做：随机化选举超时、持久化 term/vote、授票时校验日志新旧、Leader 即刻心跳。
  • 可选：PreVote、CheckQuorum、只读租约、限速与退避、快照旁路。
  • 监控：Leader 切换、选举耗时、RPC 成功率、lag、fsync 延迟、GC 停顿。

15. 面试答题模板（可直接背诵）

  • 结构化回答：
    1. 先画三角色/两计时器/两 RPC；
    2. 讲随机化超时与授票条件（term & 日志新旧）；
    3. 说三大异常：分票、分区、停顿；
    4. 给工程化：PreVote/CheckQuorum/持久化/参数建议；
    5. 收尾以监控与演练。
  • 一句话总结：
    • “Raft 通过随机化超时 + 多数派投票 + 日志新旧校验确保选举唯一且安全；生产中用 PreVote/CheckQuorum 抗抖动，并用合理的超时与监控保证稳定。”

16. 小结

  • 选举的目标是“快速且安全地产生唯一 Leader”。
  • 工程上，更看重“抗抖动能力”和“观测/演练闭环”。
  • 只有与日志复制、快照与恢复策略协同设计，Raft 集群才能在高压与异常下长期稳定运行。
