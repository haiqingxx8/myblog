分布式系统接口幂等性设计

发表于 2025-02-12 分类于 架构与中间件 阅读次数：
分布式系统接口幂等性设计，从原理到落地，详细介绍了幂等性的概念、触发源、实现方式和关键设计要点。

一、为什么要做幂等

  • 核心目标：在网络抖动、超时重试、消息至少一次投递、用户多次提交等情况下，多次执行具有同一影响（一次或零次），避免数据重复与副作用放大。
  • 常见触发源：客户端/网关自动重试、MQ 重投、定时补偿任务、第三方回调重发、运维脚本手动重放、业务并发双击。
  • 适用边界：幂等保证的是“结果不会重复副作用”，不等于强一致；与“Exactly Once”不同，多采用“至多一次/至少一次 + 幂等处理”组合实现。

二、概念澄清（语义与接口）

  • 语义幂等：以业务意义衡量重复执行是否产生额外影响（如设置昵称、扣库存）。
  • 接口幂等：以接口输入视角衡量“同一请求标识”重复提交的可控性与可预测输出。
  • HTTP 幂等语义：GET/HEAD/OPTIONS/DELETE 理论上语义幂等，PUT 幂等，POST 默认非幂等（可用自定义幂等键实现幂等）。
  • 幂等与可重入：幂等强调重复调用结果一致；可重入强调在执行中可再次进入且不破坏状态。

三、幂等实现总览（选型地图）

  • 客户端/网关层：Idempotency-Key、去重令牌、前端禁用二次点击、表单一次性 Token。
  • 服务层（最常用）：业务幂等键 + 幂等表/唯一约束、Redis 去重键、状态机防重复提交、乐观锁（version）。
  • 存储层：UPSERT/ON CONFLICT DO NOTHING、唯一索引、幂等影子表（状态机记录）。
  • 消息链路：Outbox + 消费端幂等、去重表/去重缓存、幂等消费窗口（最近 N 消息）。
  • 回调/对外：回调去重（回调单号唯一）、签名防重放、回调幂等表。

四、幂等关键设计要点

  • 幂等键选择：能唯一标识一次业务意图，如业务订单号、业务请求号、支付商户单号、三元组（userId+bizType+clientReqId）。
  • TTL 与持久：短事务可用缓存 TTL；金融级/对账要求需持久化幂等表，至少保留账期内查询能力。
  • 状态机防重：统一状态流转 INIT→PROCESSING→SUCCESS/FAIL（含可重试/不可重试），重复请求按状态返回一致响应。
  • 并发控制：相同幂等键串行化处理（本地互斥/分布式锁/单分片绑定）；避免“写后读未提交”引发误判。
  • 失败与重放：不可重试失败要正确落盘并返回可诊断码；可重试失败提供安全重放入口（需保证仍幂等）。

五、服务端常用幂等模式

  1. 幂等表 + 唯一约束
  • 设计：幂等记录表（idempotency_record），唯一索引(idem_key)，保存请求体摘要、状态、最终响应、过期时间。
  • 流程：第一次请求写入 INIT→业务处理→SUCCESS 并回填响应；后续同 key 直接返回缓存响应。
  • 优势：强诊断、可追溯、审计友好；不足：写放大、需要清理策略。
  1. Redis 去重键（SET NX EX）
  • 设计：以 idem:{key} 做原子 SET NX EX，抢占成功才执行业务，完成后可写持久幂等表或删除键。
  • 优势：延迟低、吞吐高；不足：仅缓存级诊断能力，过期策略需谨慎，注意脚本原子性与时钟漂移。
  1. UPSERT/唯一索引兜底
  • 设计：核心数据表使用唯一索引（如 merchant_order_no），利用“插入或忽略/冲突更新”实现天然幂等。
  • 优势：最贴近真实副作用，抗并发；不足：需要良好表结构设计与异常码解析。
  1. 乐观锁（版本号）
  • 设计：按 version 更新，失败说明被处理过或并发写入冲突；结合重试达成最终一次成功。
  • 场景：余额扣减、状态推进等需要防止并发覆盖。

六、消息链路的幂等

  • Outbox：事务内写业务数据与 Outbox，再异步投递；消费者按业务键去重/状态机推进，避免重复消费副作用。
  • 至少一次投递：消费端使用去重表（唯一索引 messageId 或 bizKey+type），或 Redis 最近窗口去重。
  • 事务消息/CDC：同分布式事务文档配合使用，幂等落地仍靠“业务键+状态机/唯一约束”。

七、第三方回调幂等

  • 设计：平台回调包含平台单号、商户单号与签名；商户侧以商户单号+平台单号做唯一去重并按状态机更新。
  • 重放防护：签名时效、一次性 nonce、回调白名单；回调结果可缓存并按 key 返回一致响应。

八、典型业务案例

  1. 支付下单
  • 幂等键：merchantId + merchantOrderNo。
  • 处理：创建订单前先写幂等表 INIT→调用支付网关→落 SUCCESS；重复请求直接返回第一次响应。
  1. 发券/积分发放
  • 幂等键：userId + campaignId + requestId。
  • 处理：发放记录唯一约束，重复触发返回原记录；失败重试须保证不重复发放。
  1. 账户转账
  • 幂等键：transferId；
  • 处理：借贷分录唯一约束 + 状态机；重复请求不重复记账。

九、接口与协议设计建议

  • 明确对外幂等协议：POST 接口要求客户端提供 Idempotency-Key；服务端返回 requestId/幂等键与标准化错误码。
  • 一致响应：重复请求返回与首个成功请求相同 body/状态码（常见 200/201/409 约定其一）。
  • 幂等键承载：HTTP Header（Idempotency-Key）或 body 字段；需绑定业务主体以防撞键。
  • 幂等窗口：为可补偿业务定义幂等有效期与归档策略。

十、关键表结构与索引示例

  • 幂等记录表：
    • id (PK)
    • idem_key (UK)
    • biz_type, biz_key
    • request_hash (请求体摘要)
    • status (INIT/PROCESSING/SUCCESS/FAIL/EXPIRED)
    • response_snapshot, error_code, error_msg
    • expire_at, created_at, updated_at
  • 消费去重表：message_id/biz_key+type 唯一索引，保留近账期数据。

十一、参考实现（伪代码片段）
A) Redis + 幂等中间件

  • 入口拦截：SET NX idem:{key} EX ttl → 成功则执行业务，否则快速返回“处理中/已完成”的统一响应。
  • 原子性：使用 Lua 将读写与过期放在一次脚本执行；避免 SET 后异常导致脏键。
  • 完成后：写持久幂等表 SUCCESS 并回填响应，亦可选择删除缓存键或留给 TTL。

B) 幂等表工作流

  • 首次请求：INSERT INIT（UK 冲突直接读已有记录）→ 业务执行成功 → UPDATE SUCCESS + response。
  • 重复请求：按 idem_key 查询记录，若 SUCCESS 直接返回快照；若 PROCESSING 返回“处理中”；若 FAIL 且可重试，走重试分支。

C) 消费端去重

  • 消费前先 UPSERT 去重表（messageId），若受影响行=0 认为重复，直接 ACK；否则执行业务并提交。

十二、与分布式锁/事务的关系

  • 幂等 ≠ 分布式锁：锁用于互斥并发访问；幂等用于对重复执行的副作用约束，二者常组合使用（如同 key 串行 + 幂等状态机）。
  • 幂等与事务：局部事务保证单体一致性，跨服务通常采用 Outbox/事务消息/Saga 等，幂等是补偿与重试的“安检门”。

十三、参数与容量规划

  • 幂等键 TTL：取决于业务补偿窗口（如支付 7-15 天，对账期内保留）。
  • 并发与隔离：同幂等键串行；跨键可并行；热点键需要分桶或排队。
  • 存储容量：幂等表与去重表按 QPS×保留时间估算行数，设置分区或归档任务。

十四、监控与告警

  • 指标：
    • 幂等命中率、重复请求比例、PROCESSING 超时率
    • 首次成功时间、平均/99 线处理时长
    • 去重表写入失败/冲突率、Redis 脚本耗时
    • 回调重放次数、签名校验失败率
  • 告警：幂等命中率异常升高、PROCESSING 过久、去重表冲突激增、回调失败/重放超阈值。

十五、故障演练与排障

  • 场景：
    • 客户端超时重试×3；MQ 重投×N；第三方回调重放×5
    • Redis 故障/抖动；数据库唯一索引冲突风暴
    • 热点幂等键突发，排队与锁竞争
  • 定位：
    • 按 idem_key 还原全链路；比对请求体摘要与响应快照
    • 检查状态机边界条件与回滚/补偿的原子性
    • 栈顶错误与下游实际副作用是否匹配（防“写成功回滚失败”的双写）

十六、常见坑与对策

  • 仅缓存不落盘：导致跨天对账无法追溯 → 关键链路持久化幂等记录
  • 幂等键选择不当：撞键或粒度过粗 → 绑定业务主体，纳入 bizType
  • 先写业务后写幂等表：首写成功但幂等记录失败 → 使用事务或最终一致回补
  • 状态机漏状态/竞态：PROCESSING 超时未兜底 → 增加守护清理与重放队列

十七、面试高频问答（速记）

  • 问：POST 如何实现幂等？
    • 答：客户端携带 Idempotency-Key；服务端以幂等表/Redis 去重；首个成功响应快照复用返回；冲突时返回一致响应/状态码。
  • 问：消息至少一次如何防重？
    • 答：消费端 UPSERT 去重表或 Redis 去重窗口；业务以唯一约束/状态机确保副作用只发生一次。
  • 问：与分布式锁区别？
    • 答：锁解决并发互斥，幂等解决重复副作用，常组合使用。
  • 问：如何选择 TTL？
    • 答：按补偿/对账窗口，金融级按账期，活动类按活动期+安全缓冲。

十八、生产落地清单（可直接复用）

  • 统一幂等键规范与生成器（Header/Body 支持）
  • 幂等中间件（拦截器/Filter）+ Redis Lua + 持久幂等表
  • 关键表唯一索引与 UPSERT 策略
  • 消费去重表与埋点指标
  • 守护任务：PROCESSING 清理、过期归档
  • 演练脚本：重试/重放/热点压测

十九、小结

  • 幂等是分布式系统“可重试”的基石，建议以“业务幂等键 + 状态机 + 唯一约束/去重缓存”为主线，结合监控与演练形成闭环。
