微服务限流算法：从原理到生产落地

发表于 2025-03-15 分类于 架构 ， 微服务 阅读次数：
系统梳理微服务限流算法（固定窗口/滑动窗口/漏桶/令牌桶/并发数/GCRA），给出分布式与本地双层方案、工程参数、监控与演练清单，附带真实生产案例与面试答题模板。

1. 背景与目标

  • 背景：微服务在高并发/突发流量、下游抖动、资源预算受限时需要“有损但稳定”的保护能力。
  • 目标：保护系统可用性（放弃少数请求换整体 SLA），公平地分配资源，避免雪崩与排队爆炸。

2. 常见限流算法对比（速览）

  • 固定窗口计数：按整秒/整分钟计数，简单但边界抖动大（窗口切换瞬间可双倍）。
  • 滑动窗口（计数/日志/GCRA）：在较细粒度窗口内滚动统计，抖动小；日志法精度高成本高，计数法折中。
  • 漏桶（Leaky Bucket）：恒定速率出水，平滑突发，排队可控；适合网关和出口整形。
  • 令牌桶（Token Bucket）：令牌按速率生成，允许突发（burst），更贴近业务体验；常用在入口与服务内限流。
  • 并发数限制（In-Flight）：限制同时处理中请求数量，避免线程/连接被打满；结合排队超时更有效。

3. 设计要点（工程化）

  • 多维 Key：API 路径/方法、租户/用户、IP、来源端（web/app）、环境、优先级。
  • 层级限额：全局 → 租户 → 应用 → 用户 → 接口，多级配额叠加与兜底。
  • 策略模式：拒绝（Fail Fast）、排队（Queueing，设置最大等待时间）、降级（返回缓存/兜底结果）、熔断联动。
  • 突发与预热：设置 burst（N 倍瞬时）、Warm-Up（逐步放开），平衡冷启动。
  • 动态配置与灰度：集中治理平台（配置下发/灰度生效/回滚），避免发版改阈值。
  • 本地+集中：本地快速判定（Guava/Resilience4j）+ Redis/网关集中配额，防止单点失真。

4. 分布式限流（Redis/Lua 与网关）

  • Redis + Lua 原子脚本（滑动窗口计数/令牌桶），支持多实例一致视图。
  • 网关/代理层：Nginx/Envoy/Istio 限流插件，适合全局与租户级；服务内用于接口/线程池保护。
  • 误差控制：一致性哈希将 key 分片到固定节点；时间同步容忍；跨 AZ/Region 设置更大窗口与冗余。

6. 生产案例（真实场景）

  • 案例 A：电商“秒杀”入口
    • 方案：网关 Nginx 漏桶整形 5k rps → Redis 令牌桶全局 2k rps → 服务内本地 1k rps + 并发 500。
    • 要点：库存扣减异步化/预扣，丢弃即刻返回友好文案；黑白名单与人机识别。
  • 案例 B：多租户 SaaS API 配额
    • 方案：按租户/应用 key 维度滑动窗口每日/每秒双限；管理端动态调配配额与突发。
    • 要点：月底冲刺期扩大突发，超额进入队列并限制最大等待 100ms。
  • 案例 C：媒体转码服务
    • 方案：以“并发任务数”限流，队列采用短等待 + 超时丢弃；后台按权重分配 GPU 槽位。
    • 要点：分组优先级，保障付费用户；对长任务设置最大执行时长与取消。

7. 参数建议

  • 窗口与速率：基于 p95/p99 延迟与资源容量回推 rps；burst 不超过稳态的 1~3 倍。
  • 排队：最大等待 50~200ms；超过即失败，避免排队雪崩。
  • 并发：线程池/连接池上限 < 实例 CPU×2~4；忙等改为可中断等待。
  • Redis：key TTL = 窗口；热点 key 分片；使用 pipeline/cluster。

8. 监控与告警

  • 通过/拒绝/排队比、等待时间直方图、超时丢弃数、限流命中率。
  • 热门 Key 和租户前十；Redis 脚本 QPS/时延、失败率；本地限流的触发频率。
  • 与业务：订单/支付成功率、转化率变化，避免过度限流误伤。

9. 演练与排障

  • 演练：突发 5× 流量、Redis 降级/网络抖动、单 AZ 故障，观察系统是否平滑退化。
  • 排障：
    • 过度限流：阈值过低或未区分租户；检查热门 key 与 burst。
    • 排队爆炸：等待时间上限过长；改为快速失败或分层限流。
    • 不生效：本地/集中配置不一致；核对时钟与哈希分片。

10. 面试高频问答（要点）

  • 问：令牌桶 vs 漏桶？
    • 答：令牌桶允许突发、用户体验好；漏桶恒定速率、平滑严谨，适合出口整形与网关。
  • 问：如何做分布式限流？
    • 答：Redis+Lua 原子计数/令牌桶；或网关/Istio 全局配额；本地+集中双层校验减少误差。
  • 问：如何避免排队雪崩？
    • 答：限制最大等待时间与队列长度，Fail Fast，结合熔断与降级；优先级与租户隔离。
  • 问：如何设置阈值？
    • 答：基于容量压测 + 延迟分布，按 SLA 回推 rps；给突发与灰度空间。

11. 总结

  • 限流的本质是有损控制下的稳定性与公平性。
  • 生产落地优先“本地快速判定 + 集中一致视图”，配合降级/熔断与动态治理。
  • 以监控/演练/对账（日志抽样）闭环，持续校准阈值，才能在成本内获得最佳体验。
